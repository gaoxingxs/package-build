# APISIX Offline Build

This repository provides a GitHub Actions workflow to build Apache APISIX from source and package it for offline installation on CentOS 7 compatible systems.

## What it produces

For each target architecture, the workflow uploads:

- a self-contained RPM under `rpm/`
- a local offline repository bundle `*-repo.tar.gz`
- a plain root filesystem tarball `*.tar.gz`
- `INSTALL.txt`
- `build-dependencies.txt`
- runtime dependency reports under `reports/`
- `SHA256SUMS`
- `build-info.txt`

The RPM bundles:

- `/usr/local/openresty`
- `/usr/local/apisix`
- `/usr/local/share/lua` and Lua module directories installed by LuaRocks
- `/usr/bin/apisix`
- `/usr/lib/systemd/system/apisix.service`

It also copies APISIX Lua dependencies into `/usr/local/apisix/deps`, which is where the installed `apisix` CLI expects them.

## Workflow inputs

Workflow file: `.github/workflows/apisix-offline-build.yml`

- `apisix_version`: APISIX tag to build. Default: `3.13.0`
- `target_arch`: `amd64`, `arm64`, or `both`
- `openresty_version`: bundled OpenResty version. Default: `1.21.4.3`
- `luarocks_version`: build-time LuaRocks version. Default: `3.11.1`
- `rpm_release`: RPM release field. Default: `1`

## How it works

1. Start a CentOS 7 userspace container for the selected architecture.
2. Compile OpenResty from source.
3. Download APISIX source from the GitHub tag tarball.
4. Install LuaRocks and run `make deps` plus `make install`.
5. Copy the generated `deps` tree into `/usr/local/apisix/deps`.
6. Stage the final filesystem layout.
7. Build an RPM and an offline repository tarball.

## Install offline

### Option 1: install the single RPM

```bash
yum install -y ./apisix-offline-<version>-<release>.el7.<arch>.rpm
```

If `yum` is not available:

```bash
rpm -Uvh ./apisix-offline-<version>-<release>.el7.<arch>.rpm
```

### Option 2: install from the repo bundle

```bash
tar -xzf apisix-offline-<version>-el7-<arch>-repo.tar.gz
cd repo
./install.sh
```

## Start the service

```bash
systemctl daemon-reload
systemctl enable --now apisix
```

Or manually:

```bash
apisix init
apisix start
```

The uploaded `INSTALL.txt` in each artifact directory mirrors the exact artifact names generated by the workflow and is the safest source for copy-paste install commands.

## Important constraints

- The workflow targets CentOS 7 compatible glibc and userland by building inside a CentOS 7 container.
- `arm64` builds run through QEMU in GitHub Actions and are slower than `amd64`.
- `arm64` also depends on the availability of a usable `arm64v8/centos:7` container image in Docker Hub or your mirrored registry.
- The package is offline-installable, but APISIX still needs its runtime configuration and an external config source.
- `etcd` is not bundled. Use external etcd, or configure APISIX standalone mode.
- The runtime is built from source in this repository workflow. Apache APISIX documentation recommends `apisix-build-tools` when you need platform-specific runtime packaging at larger scale.
